## Session protocol

Mirrors the improvement-cycle protocol, adapted for the rebuild track. The rebuild branch runs independently and does not check grant dates.

1. **Setup**: `git checkout frontend-v2 && git pull origin frontend-v2`. Configure git identity: `git config user.email 'rebuild@cultivos.agency'` and `git config user.name 'cultivOS rebuild agent'`.
2. **Read the backlog** (`.rebuild/backlog.md`). Find the first `[ ]` item.
3. **Pre-flight**: if the selected item is R0, there is no prior scaffold to validate — skip to step 4. Otherwise, `cd frontend-v2 && npm ci && npm run build`. Must succeed before you touch anything. If the pre-flight build fails, append `BLOCKED: pre-flight build failed — <error summary>` to `.rebuild/overnight-log.md` (create if missing), commit as `overnight: log BLOCKED for R<n>`, push, exit.
4. **Execute** the item's instructions verbatim. No scope creep.
5. **Post-flight**: `cd frontend-v2 && npm run build` again. Must succeed. (R0 does this as part of its own instructions.)
6. **Status log**: append to `.rebuild/overnight-log.md`: `YYYY-MM-DD HH:MM Toronto — R<n> — DONE — <short summary>`. No commit hash.
7. **Mark done**: flip the item's checkbox in `.rebuild/backlog.md`.
8. **Stage everything**: `git add` your changes + both `.rebuild` files.
9. **One commit** with the exact message from the item spec.
10. **Push**: `git push origin frontend-v2`. Never push to `main`. Never force-push.
11. **Exit**.

### BLOCKED flow

1. If you have local changes and cannot proceed, `git stash` them.
2. Append `YYYY-MM-DD HH:MM Toronto — R<n> — BLOCKED — <reason>` to `.rebuild/overnight-log.md`.
3. Stage only `.rebuild/overnight-log.md`. Commit with `overnight: log BLOCKED for R<n>`.
4. Push and exit. Do NOT mark the item `[x]`.

### IDLE flow (queue empty)

1. Append `YYYY-MM-DD HH:MM Toronto — — IDLE — rebuild queue empty` to `.rebuild/overnight-log.md`.
2. Commit `overnight: log IDLE — rebuild queue empty`.
3. Push and exit.

## Safety clamps

- Max **one commit** per session. Ever.
- Never push to `main`. Never force-push any branch.
- Never delete files unless the item explicitly says 'delete'.
- Never touch files outside the repo working tree.
- Never touch `src/cultivos/` or the old repo-root `frontend/` directory — those belong to main and improvement-cycle tracks. Your world is `frontend-v2/` + `.rebuild/`.
- Never run destructive git commands unless the item explicitly says to.
- If you find yourself about to do something not in the item spec, stop and log BLOCKED.
- If `npm run build` fails and you cannot resolve it inside the item's scope, log BLOCKED.
- Do not cite external blogs, "industry best practices," or framework docs as justification in comments or commit messages. Our way is the standard.
