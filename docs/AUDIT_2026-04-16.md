# cultivOS — Full-stack Audit

**Date**: 2026-04-16
**Auditor**: AutoAgent (opus-4-7)
**Scope**: Architecture, security, data, frontend, tests, ops, FODECIJAL grant readiness
**Method**: Static analysis of the repository at HEAD (`e231903`). No runtime probing, no deploy verification.

---

## 1. Executive Summary

cultivOS has grown from a CRUD demo into a 74-router, 218-endpoint platform with 3,627 passing tests and 86 frontend pages. The velocity is impressive — and visible in the grant narrative.

However, the platform is **not production-ready** in the security or ops sense:

- **Only 4 of 218 routes require authentication.** ~98% of endpoints are effectively public.
- **JWT is hand-rolled**, not a library. Acceptable for demo; risky for production.
- **No CI.** `.github/workflows/` is empty. A red commit can reach `main` unnoticed.
- **No database migrations.** Schema changes rely on `Base.metadata.create_all` and destructive reseeding.
- **Frontend has zero i18n infrastructure.** Spanish text is hardcoded in 86 HTML files — no ES/EN toggle mechanism anywhere.
- **`NORTH_STAR.md` metrics drift.** Reports 2,539 tests (actual 3,627), 64 pages (actual 86), 50 API route files (actual 74). Grant reviewers will see these numbers.

For FODECIJAL (deadline May 14, 2026) the **demo tour works** and the narrative PDF auto-updates from live DB state. The highest-value pre-submission fixes are (1) update `NORTH_STAR.md` and the PDF stats before the reviewer reads them, (2) make `whatsapp-demo.js` call the real intelligence endpoint, and (3) document the auth story honestly — TRL 4/5 does not require production-grade auth, but the gap should be acknowledged in the grant narrative rather than hidden.

See **§9 Top 10 Prioritized Improvements** for the actionable list.

### 2.3 DB schema

30 ORM classes in one `db/models.py`. Key entities: `Farm`, `Field`, `User`, `Cooperative`, `FlightLog`, `HealthScore`, `NDVIResult`, `ThermalResult`, `SoilAnalysis`, `MicrobiomeRecord`, `TreatmentRecord`, `FarmerObservation`, `TEKAdoption`, `Alert`, `AlertConfig`, `AlertLog`, `CarbonBaseline`, `PredictionSnapshot`, `FarmerFeedback`, `HarvestRecord`, plus knowledge tables (`AncestralMethod`, `Fertilizer`, `Disease`, `CropType`, `CropVariety`, `AgronomistTip`, `FarmerVocabulary`).

**Observation.** 30-class single-file ORM is still readable but starting to sprawl. Splitting by bounded context (farm/, knowledge/, alerts/, telemetry/) would cost 30 min and pay back every future refactor.

### 2.4 Layering

`api → services → utils/db` is followed in spot-checks. `services/reports.py` at the top level (not nested under a domain folder) is a small inconsistency. No obvious circular imports.

- Routers registered from a central `api/__init__.py` `all_routers` list — clean pattern.
- Custom middleware: `SecurityHeadersMiddleware` (CSP, HSTS, XFO, nosniff, XSS), `RequestLoggingMiddleware`, CORS (env-configurable), structured error envelope for 400/401/403/404/422/500.
- OpenAPI tags defined for 50+ categories — Swagger UI is rich.
- Frontend served as `StaticFiles` mount at `/` — zero build step.

## 3. Security

### 3.1 Authentication coverage

**Critical finding.** Grep result:

```
Depends(get_current_user)  → 4 occurrences across the 74 API files
Depends(require_role(...)) → 0 occurrences
Files that import auth     → 3 non-trivial (farms, intel, demo)
```

**218 routes exist. ≤4 require authentication.** Every NDVI analysis, soil record, treatment recommendation, cooperative stat, carbon calculation, and intel endpoint is callable without a token. The `/api/farms` route uses `get_optional_user` to *filter* by `farm_id` when a token is present — but the route returns all farms when no token is present.

**Posture assessment.** For a TRL 4 research demo this is tolerable. For any deployment facing real farmers, it is an unacceptable risk: any actor on the internet can exfiltrate the full farm/field/soil dataset or spam treatment recommendations.

**Pattern already exists.** `auth.py` defines `get_current_user`, `get_optional_user`, and a role-gating helper `require_role(*roles)`. The primitives are in place — they are simply not applied.

### 3.2 JWT implementation

`src/cultivos/auth.py` implements JWT **from scratch** using `hashlib.sha256` + `hmac` + base64 JSON:

```python
def create_access_token(...):   # HS256 by hand
def decode_access_token(...):   # base64 split, hmac.compare_digest
```

No `pyjwt`, no `authlib`, no `python-jose` in `requirements.txt`. The impl does use `hmac.compare_digest` (constant-time compare) and does guard on `settings.jwt_secret_key` length ≥ 16. It does **not**:

- Enforce `exp` claim validation (need to verify in `decode_access_token` body — check manually).
- Support key rotation (`kid`).
- Use a vetted library path.

**Recommendation.** Swap for `PyJWT` (two lines changed) before any public launch. Keep the interface identical.

### 3.3 Password hashing

`hash_password()` uses PBKDF2-HMAC-SHA256 with 100,000 iterations and 16-byte salt. OK for 2026 but weaker than the current recommendation (argon2id). `passlib[argon2]` would be a one-liner upgrade.

### 3.4 Ownership enforcement

Only `farms.py:120` has a real ownership check:

```python
if user and hasattr(user, 'role') and user.role == "farmer" and user.farm_id is not None:
    query = query.filter(Farm.id == user.farm_id)
```

Every other route that takes a `farm_id` path parameter does **not** verify the caller owns that farm. A farmer token can still fetch any other farm's data by URL guessing.

### 3.5 Self-registration

`POST /api/auth/register` blocks admin self-registration (line 51) but accepts any `farm_id` the caller provides. Backlog item **S3** already tracks this. No other registration hardening (email verification, rate limit, CAPTCHA).

### 3.6 Secrets

- No hardcoded API keys found in `src/` (grep for `sk-`, `AKIA`, `ghp_` returned 0 results).
- `.env.example` is the only dotfile checked in. `.env` is gitignored.
- `jwt_secret_key` defaults to `""` in config — server will refuse to sign tokens if unset (graceful).
- **Risk.** Railway secret rotation story is undocumented — rotating `JWT_SECRET_KEY` will invalidate every outstanding token with no refresh-token mechanism.

### 3.7 Transport & headers

- Full CSP with `default-src 'self'` and explicit CDN allowlist.
- HSTS (`max-age=63072000`), X-Frame-Options DENY, nosniff, Referrer-Policy strict-origin-when-cross-origin.
- CORS `allow_credentials=True` with env-configurable origins — default `localhost:3000, localhost:8000`.

**Gap.** No CSRF middleware. Cookie-based auth isn't used (tokens go in `Authorization` header), so CSRF risk is low — but any future form POST without the token in header would be exposed.

### 3.8 Rate limiting

**None.** No `slowapi`, no FastAPI limiter. Any route can be hit in a tight loop. For the grant demo this does not matter; for production it is a launch blocker.

### 3.9 Input validation

Pydantic v2 handles most body validation. Deprecation warning visible: `models/user.py:18` uses class-based Config instead of `ConfigDict` — pydantic v3 migration debt.

## 4. Data Integrity

### 4.1 Migrations

**No migration tool.** No `alembic/`, no `migrations/`, no versioned schema. `src/cultivos/db/session.py:19` calls `Base.metadata.create_all` at startup. Every schema change assumes a fresh DB or a compatible additive change.

**Consequences.**
- Adding a NOT NULL column to an existing table silently fails on existing rows (SQLAlchemy `create_all` doesn't alter).
- Renames require manual SQL or a reseed.
- Railway's persistent volume holds a live DB — there's no playbook for evolving it.

**Recommendation.** Add `alembic` now, before the grant submission. The effort is 1–2 hours: `alembic init`, autogenerate the baseline revision, commit. Zero downstream impact.

### 4.2 Seed data

`db/seeds.py` is **1,354 lines** — 7+ seeding functions run in `_lifespan()` when `DB_URL` is not `:memory:`. Each `seed_*` function is idempotent (checks `if count_existing > 0: return 0`) so re-seeding is safe.

**Observation.** Seeds.py combines fixture data (fertilizers, diseases, crops, ancestral methods, vocabulary, agronomist tips) with what appears to be demo/portfolio data. For an empty Railway DB this is correct. For a real farmer-onboarding scenario, demo farms should be gated behind a flag.

### 4.3 Repository hygiene

- `cultivos.db` is **not** tracked (`git ls-files cultivos.db` → empty). `.gitignore` correctly excludes `*.db`, `*.db-shm`, `*.db-wal`.
- `.autoagent/` gitignored — meta-agent artifacts stay out of the project repo. ✓
- `imagery/`, `uploads/`, `*.tiff`, `*.dng` gitignored — drone imagery won't pollute the repo. ✓
- **Minor.** `src/cultivos/__pycache__/auth.cpython-311 2.pyc` (with the `" 2"` suffix) suggests an iCloud Drive duplicate conflict. Harmless but should be cleaned.

### 4.4 Demo vs prod drift

The FODECIJAL PDF (`src/cultivos/api/fodecijal_report.py`) computes stats from the live DB (`compute_platform_stats()`), so numbers auto-track actual state. **Good.** The risk vector is `.autoagent/NORTH_STAR.md` — its manual metrics table is stale (see §7.1).

## 5. Frontend

### 5.1 Layout

```
frontend/
  86 .html pages
  85 .js modules
  styles.css (single file, 27 media queries)
```

No build step, no bundler, no TypeScript, no framework. This is deliberate (CLAUDE.md) and **correct** for the current stage — zero deploy complexity, easy to read.

### 5.2 Size distribution

Top 5 JS files by bytes:

| File | Bytes |
|---|---:|
| `field.js` | 106,132 |
| `app.js` | 51,380 |
| `intel.js` | 31,600 |
| `intelligence-page.js` | 15,944 |
| `management.js` | 12,933 |

`field.js` at 106KB is a monolith — almost certainly handles multiple views and has internal duplication. Worth splitting when time permits (low urgency; not grant-critical).

### 5.3 Code duplication

- **31 raw `fetch('/api/...')` calls** across different JS files. No shared API client module. Every page re-implements error handling, loading states, and JSON parsing.
- **Impact.** Adding a global `Authorization` header for the auth work (§3) means editing 31+ sites instead of one helper. This is the single biggest frontend tax.

**Recommendation.** Introduce `frontend/api.js` with `api.get(path)` / `api.post(path, body)` that:
- reads the bearer token from `localStorage`
- handles 401 → redirect to login
- parses JSON with consistent error envelope

Cost: 1 hour. Savings: recurring.

### 5.4 Internationalization

**Zero i18n infrastructure.** Grep for `data-i18n`, `data-es`, `toggleLang`, `setLocale`, `currentLang` returned **0 matches** across the frontend tree. Spanish text is baked directly into HTML. The `AUDIENCE: mixed` / Spanish-first policy in CLAUDE.md is honored *by writing Spanish*, not by supporting a toggle.

**Consequence.** Any future Ontario deployment requires either (a) duplicating every page with English strings, or (b) retrofitting i18n across 86 HTML files. Option (a) is faster for the grant phase; option (b) is the eventual cost.

**Recommendation for the grant.** Document the Ontario-expansion path as "planned Q3 2026" rather than in-progress.

### 5.5 Accessibility

Not audited in depth this pass. Quick checks:
- `lang="es"` attribute present on HTML elements: yes (every page).
- `aria-label` / `role=` usage: not counted exhaustively. Recommend a dedicated Playwright accessibility pass with `axe-core` before any public launch.

### 5.6 Mobile responsiveness

`styles.css` contains **27 `@media` queries** with breakpoints at 480/640/768/1024. Backlog item **D5** already flags that a pass at 375px width is needed — this is the iPhone SE size FODECIJAL reviewers might test on.

### 5.7 Third-party assets

CSP allows `unpkg.com` and `cdn.jsdelivr.net`. Actual usage not inventoried (likely Chart.js, Leaflet, maplibre). Every external dep is a supply-chain surface — pinning specific versions in the HTML `<script src>` URLs would be safer.

## 6. Tests

### 6.1 Coverage

- **3,627 tests collected** in 1.68s across 326 test files (`pytest --collect-only -q`).
- Test suite configured for in-memory SQLite (`conftest.py` sets `DB_URL` to `sqlite:///:memory:` before import).
- Fixtures hand out a fresh `Session` per test.

### 6.2 Test quality

- **Real DB, not mocks.** Only 12 of 326 test files use `unittest.mock` / `monkeypatch`. Nearly all tests touch the real ORM — strongest signal that what passes in tests will pass in prod.
- **Deprecation warning.** `models/user.py:18` uses class-based `Config` — Pydantic v3 will reject this. Known cleanup debt.

### 6.3 E2E / Playwright

- `tests/test_e2e_farmer_journey.py` exists but is **pytest-based**, not Playwright.
- No `playwright.config.ts`, no `playwright/` directory, no `@playwright/test` dev dep.
- `skills/playwright.md` referenced in PROMPT.md implies Playwright IS used — but by the agent, not by pytest.

**Gap.** No automated browser test runs in CI (because there is no CI — see §7). The 86 frontend pages could regress silently.

### 6.4 Flakiness

Not measured. Pure-unit tests with in-memory SQLite should be deterministic; flake risk is near zero for the current suite.

## 7. Ops

### 7.1 Deployment

- **Dockerfile.** Two-stage `python:3.12-slim` build, installs requirements in builder, copies `src/`, `frontend/`, `scripts/`. Runs `uvicorn cultivos.app:create_app --factory --host 0.0.0.0 --port $PORT`. Clean and minimal.
- **`railway.toml`.** Uses `DOCKERFILE` builder, `restartPolicyType = "ON_FAILURE"`. No explicit build command, no healthcheck, no env pin.
- **No healthcheck** configured on Railway — Railway will only restart on process crash, not on a wedged app.

### 7.2 CI

**Empty `.github/workflows/`.** No GitHub Actions, no pre-commit, no branch protection workflow. A commit that breaks tests or imports can land on `main` and deploy to Railway unnoticed. Test-count drift in §7.1 NORTH_STAR is partly a symptom of this.

**Recommendation.** One `ci.yml` that runs `pytest tests/ -q` on every push. ~20 lines of YAML. Blocker-severity gap for a grant-funded project.

### 7.3 Observability

- **12 `logger.*` call sites in `src/`.** Very thin. Most endpoints log nothing.
- `RequestLoggingMiddleware` at least captures method, path, status, duration per request.
- No structured logging (JSON), no log shipping, no error aggregator (Sentry/Rollbar). Any production exception disappears unless someone tails Railway logs.

**Recommendation.** Add Sentry via a ~5-line init. Free tier covers grant-phase volume. Without it, a prod crash is invisible.

### 7.4 Secrets & rotation

- Keys documented in `.env.example`. Status in `.autoagent/KEYS_NEEDED.md` (per agent protocol).
- No rotation playbook. Rotating `JWT_SECRET_KEY` invalidates all tokens (no refresh-token flow).

## 10. Overall Verdict

**Demo-ready: yes.** **Production-ready: no, and that's fine for TRL 4→5.**

The platform is substantively built, well-tested, and narratively strong. The three gaps that most threaten the grant submission are stale metrics in `NORTH_STAR.md`, the static WhatsApp demo, and the missing CI workflow. All three are ≤1-day fixes.

The larger gaps — auth coverage, migrations, observability — are real but are defensible at TRL 4/5 *if acknowledged*. The worst outcome would be a reviewer discovering them before the narrative does. Better to name "auth and migrations are on the Q3 hardening roadmap" in the application itself than to let a technical reviewer find the gaps first.

Priority order for the next 28 days: **P0 items before May 1, P1 items before May 10, P2 items post-submission.**
